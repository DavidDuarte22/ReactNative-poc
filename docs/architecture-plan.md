## 2. The Key Architectural Decision

Before Phase 1 starts there is **one open question** that changes the implementation path significantly:

> **Can expo-brownfield own the native host setup, or do we keep `TescoRNHost.mm`?**

This determines whether we follow Path A (expo-brownfield) or Path B (manual). Both paths produce the same end state: an XCFramework consumed by Tesco's Tuist/SPM project. The difference is who builds and maintains the plumbing.

## 4. The Two Paths

### Path A — expo-brownfield

**What it is:** Expo's official tooling for the isolated brownfield pattern. Ships with Expo SDK 55 (RN 0.83). The CLI produces an XCFramework from your Expo project without writing any custom build scripts.

**What expo-brownfield provides:**

| Feature | How |
|---|---|
| XCFramework build | `npx expo-brownfield build:ios` |
| Hermes packaging | Outputs `hermesvm.xcframework` alongside your app framework |
| SPM distribution | Documented, supported natively |
| RN → Native messaging | `Brownfield.sendMessage(type, payload)` |
| Native → RN messaging | `BrownfieldMessaging.sendMessage(type, payload)` |
| Shared state sync | `useSharedState(key)` / `setSharedStateValue(key, value)` |
| Navigation | `Brownfield.popToNative()` |

**What the communication layer looks like under Path A:**

```swift
// Native side (Tesco's app) — no RN imports
BrownfieldMessaging.sendMessage("cartUpdated", data: ["count": 3, "items": [...]])
BrownfieldMessaging.addListener("addToCart") { message in
    CartManager.shared.add(CartItem(from: message.data))
}
```

```ts
// RN JS side
import { Brownfield } from 'expo-brownfield'

Brownfield.sendMessage('addToCart', { skuId: 'abc', price: 4.99 })
Brownfield.addMessageListener('cartUpdated', (data) => setCartCount(data.count))

// Or with shared state:
const [cartCount] = useSharedState<number>('cart.count')
```

**The critical unknown — does expo-brownfield handle `TescoRNHost.mm`?**

Our `TescoRNHost.mm` contains RN 0.84-specific fixes that took significant time to get right:

```objc
// These must all be present in expo-brownfield's generated setup:
ReactNativeFeatureFlags::dangerouslyForceOverride(...)      // RN 0.84 feature flags
RCTNetworking with custom HTTP/data/file handlers           // bridgeless networking
DefaultTurboModules::getTurboModule(name, jsInvoker)        // NativeMicrotasksCxx
ExpoModulesAdapter.setupWithRuntime — JavaScriptActor       // JS thread constraint
config.jsRuntimeConfiguratorDelegate = self                 // Hermes factory
```

If expo-brownfield's prebuild generates equivalent setup correctly → adopt Path A, delete `TescoRNHost.mm`.
If it misses any of these → stay on Path B, optionally use expo-brownfield's JS APIs only.

**RN version gap:** Expo SDK 55 ships with RN 0.83. We are on RN 0.84 (one minor version ahead). The `expo-brownfield-target` peer dependency is `expo: *` (wildcard) — not pinned to a specific SDK. The mismatch is minor and likely workable in practice.

**Important caveat:** expo-brownfield requires restructuring the RN project to use Expo's CNG workflow (`app.json` + `expo prebuild`). This is not a large change given we already have `expo-modules-core` installed, but it means the native project is generated — not manually maintained.

### Path B — Manual provider pattern

**What it is:** We own the entire XCFramework build pipeline and communication layer. `TescoRNHost.mm` stays. Custom Expo Module wrappers delegate to typed Swift protocols.

**When to use:** If expo-brownfield's generated setup does not handle the RN 0.84 New Architecture requirements correctly. Or if the CNG workflow constraint (generated native project) is incompatible with how the project needs to evolve.

**What the communication layer looks like under Path B:**

```swift
// TescoRNContract — shared Swift protocols (SPM source package)
public protocol TescoCartProvider: AnyObject {
    func addItem(_ item: [String: Any], completion: @escaping (Error?) -> Void)
    func onCartChanged(_ handler: @escaping ([String: Any]) -> Void) -> CartObserverToken
}

// TescoCartModule.swift — Expo Module wrapper (inside XCFramework)
public class TescoCartModule: Module {
    public func definition() -> ModuleDefinition {
        Name("TescoCart")
        AsyncFunction("addItem") { (item: CartItemRecord, promise: Promise) in
            TescoRNRegistry.cartProvider?.addItem(item.toDictionary()) { error in
                error != nil ? promise.reject(error!) : promise.resolve()
            }
        }
        Events("onCartChanged")
        OnCreate {
            TescoRNRegistry.cartProvider?.onCartChanged { [weak self] state in
                self?.sendEvent("onCartChanged", state)
            }
        }
    }
}
```

**Trade-offs vs Path A:**

| | Path A (expo-brownfield) | Path B (manual) |
|---|---|---|
| **Build tooling** | CLI automated | Custom scripts to write + maintain |
| **Native host** | Generated by expo-brownfield | Owned: `TescoRNHost.mm` |
| **Communication type safety** | Stringly-typed message bus | Typed Swift protocols |
| **Boilerplate per new feature** | ~5 lines JS + ~5 lines native | ~30 lines Swift + protocol definition |
| **Dependency risk** | expo-brownfield maturity (SDK 55, still early) | No extra dependency |
| **Control** | Lower — CNG generates native project | Full control |
| **Android parity** | Built in — same pattern works | Separate Android work |

## 5. Native Side — Tuist + SPM

Tuist generates the Xcode project from `Project.swift` manifests. SPM runs under Tuist to resolve dependencies. Both XCFrameworks are consumed as SPM binary targets — no CocoaPods anywhere in Tesco's project.

```swift
// Tuist Project.swift — Tesco's app (same for both paths)
let project = Project(
    name: "TescoApp",
    packages: [
        .remote(url: "https://github.com/tesco/tesco-rn-module/releases/.../TescoRNModule.xcframework.zip",
                requirement: .exact("1.0.0")),
        .remote(url: "https://github.com/tesco/tesco-rn-module/releases/.../hermesvm.xcframework.zip",
                requirement: .exact("1.0.0"))
    ],
    targets: [
        .target(
            name: "TescoApp",
            dependencies: [
                .package(product: "TescoRNModule"),
                .package(product: "hermesvm")
            ]
        )
    ]
)
```

**`pod install` never runs in Tesco's CI.** CocoaPods is entirely inside the RN module's own pipeline.

**SPM under Tuist:** Tuist calls `swift package resolve` internally and maps packages into the Xcode project graph. Binary SPM targets (`.binaryTarget`) are supported. Pin the Tuist version — there are known bugs with transitive binary target resolution in some versions.

## 6. New Architecture Constraints (both paths)

- **Bridgeless mode** — no `RCTBridge`. All communication via JSI. Not optional.
- **Static linking required** — `use_frameworks! :linkage => :dynamic` breaks C++ ODR. This rules out SPM's `spm_dependency` helper for adding Swift packages to the RN project.
- **One JS runtime per process** — one `RCTHost`, one Hermes instance, one bundle. All RN surfaces share it.
- **CocoaPods required inside the RN build pipeline** — to compile and statically link RN's ~50 pods before producing the XCFramework. Never visible to Tesco's project.
- **`RCT_USE_PREBUILT_RNCORE=1`** — skips recompiling RN's C++ deps in CI. Use from Phase 1 onward.

## 7. Expo Modules vs TurboModules

### Decision: Expo Modules for all bridge modules (both paths)

The PoC validated this. `TescoNativeBridgeModule.swift` is 14 lines of pure Swift — no ObjC++, no Codegen.

| Dimension | Expo Modules | TurboModules |
|---|---|---|
| **Language** | Pure Swift | ObjC++ required |
| **Codegen** | Not required | Required |
| **Boilerplate** | Minimal `definition()` DSL | Significant |
| **EventEmitter** | Built-in | Manual subclass |
| **Android parity** | Same pattern in Kotlin | Platform-specific each time |
| **TypeScript types** | `requireNativeModule<T>()` runtime-typed | Codegen compile-time typed |
| **Binary size** | +ExpoModulesCore (~3–4MB) | No extra dependency |
| **RN version lag** | Expo may trail RN by days–weeks | Always in sync |
| **Threading** | `JavaScriptActor.assumeIsolated` required in `didInitializeRuntime:` | Same constraint, more explicit |

**TurboModules only for:** RN core internals (`DefaultTurboModules::getTurboModule()`), C++ cross-platform modules, benchmarked performance-critical paths.

## 8. Communication Layer

### Both paths — three directions

```
Direction              Path A                          Path B
─────────────────────────────────────────────────────────────────────
Native → RN (once)    Initial props (both paths)       Initial props
Native → RN (live)    BrownfieldMessaging.sendMessage  Expo sendEvent()
RN → Native           Brownfield.sendMessage           Expo AsyncFunction
Escape hatch          NotificationCenter.default       NotificationCenter.default
```

### Cart scenario — Path A data flow

```
User taps "Add to Cart" in RN
    │  Brownfield.sendMessage('addToCart', { skuId: 'abc', price: 4.99 })
    ▼
BrownfieldMessaging listener (Tesco's app)
    │  CartManager.shared.add(CartItem(from: message.data))
    ▼
CartManager.notifyObservers()
    ├──► Native cart badge ✅
    ├──► Native cart screen ✅
    └──► BrownfieldMessaging.sendMessage('cartUpdated', ["count": 1])
              │
              ▼
         Brownfield.addMessageListener('cartUpdated') → setCartCount(1) ✅
```

### Cart scenario — Path B data flow

```
User taps "Add to Cart" in RN
    │  requireNativeModule('TescoCart').addItem({ skuId: 'abc', price: 4.99 })
    ▼
TescoCartModule.swift → cartProvider.addItem() → CartManager.add()
    ├──► Native cart badge ✅
    ├──► Native cart screen ✅
    └──► sendEvent("onCartChanged", ["count": 1])
              │
              ▼
         DeviceEventEmitter listener → setCartCount(1) ✅
```

## 9. Phased Delivery Plan

### Phase 0 — Current PoC *(done)*

New Architecture + Expo Modules proven end-to-end. Standalone workspace.

---

### Phase 1 — Validation Spike *(next — 1–2 weeks)*

**Goal:** Decide between Path A and Path B with code, not theory.

**Spike A — expo-brownfield validation:**

```
1. Add expo + expo-brownfield to package.json
2. Add app.json with brownfield plugin config
3. Run: npx expo prebuild
4. Inspect generated TescoRNHost equivalent:
   - Does it have RCTNetworking HTTP handlers?
   - Does it call DefaultTurboModules::getTurboModule()?
   - Does ExpoModulesAdapter setup correctly?
   - Does ReactNativeFeatureFlags override work?
5. Run: npx expo-brownfield build:ios
6. Check artifacts/ for TescoRNModule.xcframework + hermesvm.xcframework
7. Wire into a stub Tuist project via SPM binary target
8. Implement cart add via Brownfield.sendMessage
9. Confirm it works end-to-end with no pod install in Tuist CI
```

**Spike B — manual XCFramework baseline (run in parallel):**

```
1. RCT_USE_PREBUILT_RNCORE=1 pod install
2. xcodebuild archive → TescoRNModule.xcframework
3. Wire into stub Tuist project via .xcframework(path:)
4. Implement TescoCartModule.swift with CartManager stub
5. Confirm end-to-end with no pod install in Tuist CI
```

**Decision gate:** If Spike A works → Path A. If not → Path B. Run both in parallel, choose based on evidence.

---

### Phase 2 — First Real Integration *(Path determined by Phase 1)*

**Goal:** Real cart feature working across the binary boundary in Tesco's actual Tuist project.

**Path A deliverables:**
- expo-brownfield XCFramework published to GitHub releases with `Package.swift`
- `hermesvm.xcframework` published alongside
- `BrownfieldMessaging` cart add/remove/sync working
- Tesco's CI: `tuist install → tuist generate → xcodebuild` (no pod install)

**Path B deliverables:**
- `TescoRNContract` SPM source package (protocols: `TescoCartProvider`)
- `TescoRNRuntime.xcframework` published via GitHub releases
- `TescoCartModule.swift` with `CartManager` integration
- Tesco's CI: same (no pod install)

**Both paths — OTA bundle:**
- Metro bundle + Hermes bytecode pushed to CDN
- `manifest.json` with `minFrameworkVersion` compatibility check
- JS updates ship without App Store release

---

### Phase 3 — Production Hardening

- dSYM publishing alongside each XCFramework release
- Bundle ↔ framework version compatibility enforcement
- Error boundaries per RN surface in JS
- Feature flags at surface level (remote config gate before presenting any RN VC)

---

### Phase 4 — Enterprise Scale *(when needed)*

- JS code splitting — lazy-load surfaces on demand
- Multiple RN surfaces sharing one runtime
- If on Path B: evaluate migrating communication layer to expo-brownfield's `sharedState` API (by then likely stable)
- Android: expo-brownfield provides the same pattern for AAR distribution

## 10. Known Risks

| Risk | Impact | Mitigation |
|---|---|---|
| expo-brownfield misses RN 0.84 host requirements | Path A blocked | Spike A reveals this early; Path B is the fallback |
| expo-brownfield breaking changes (early-stage library) | Forced upgrades | Pin exact version; schedule upgrade windows |
| RN 0.83 vs 0.84 gap in expo-brownfield | Minor incompatibility | `expo: *` peer dep gives flexibility; validate in spike |
| Protocol/message explosion at scale | Coordination overhead | Path A: sharedState reduces per-feature work; Path B: contract package |
| Bundle ↔ Framework version mismatch | App crash on OTA update | `manifest.json` compatibility check (Phase 2) |
| Single JS runtime shared failure | One surface crash affects all | Error boundaries per surface |
| Tuist binary target bugs | Build failures | Pin Tuist version; test early |
| XCFramework ABI instability on RN upgrade | Full rebuild required | Planned upgrade windows |

## 12. Locked Decisions

| Decision | Rationale |
|---|---|
| **XCFramework + SPM binary target** | Complete CocoaPods isolation from Tesco's pipeline — both paths produce this |
| **Expo Modules for bridge layer** | Pure Swift, built-in EventEmitter, Android parity |
| **TurboModules only for RN core internals** | `DefaultTurboModules` handles these |
| **Static linking** | New Architecture C++ ABI; dynamic linking breaks ODR |
| **CocoaPods inside RN repo only** | Never runs in Tesco's CI |
| **One JS runtime, one bundle** | Cannot have two Hermes instances |
| **`RCT_USE_PREBUILT_RNCORE=1` in CI** | Eliminates recompiling RN's C++ deps |
| **Path A vs B** | **Open — resolved by Phase 1 spike** |
