## 1. Repository Layout

```
ReactNativePoC/
├── ReactNativeApp/                  ← RN implementation (owned by the platform/RN team)
│   ├── ios/
│   │   ├── ReactNativePoC.xcworkspace
│   │   ├── ReactNativePoC.xcodeproj
│   │   ├── Podfile
│   │   ├── TescoNativeBridgePod/    ← native modules (Expo Module DSL, pure Swift)
│   │   └── tescornappbrownfield/    ← framework sources (generated by expo prebuild)
│   ├── src/
│   │   └── App.tsx                  ← JS entry point
│   ├── artifacts/                   ← built XCFrameworks (committed or published externally)
│   │   ├── tescornappbrownfield.xcframework
│   │   └── hermesvm.xcframework
│   ├── index.js
│   ├── package.json
│   ├── metro.config.js
│   ├── babel.config.js
│   └── app.json
├── TescoUIKitApp/                   ← Native iOS consumer (owned by the native iOS team)
│   ├── TescoUIKitApp.xcodeproj
│   └── TescoUIKitApp/
│       ├── AppDelegate.swift
│       ├── HomeView.swift
│       └── CartState.swift
└── docs/
```

**Ownership boundary:** Everything inside `ReactNativeApp/` is the platform team's concern. The native iOS team only touches `TescoUIKitApp/` and the XCFramework reference in the project file. They never run `pod install`, never see CocoaPods, and never need Node.

## 4. Day-to-Day Development

### Running the JS development server (Metro)

```bash
cd ReactNativeApp
npx react-native start
```

Metro hot-reloads JS changes instantly. You do not need to rebuild the XCFramework for JS-only changes during development.

### Running the RN app on the simulator (standalone, during development)

```bash
cd ReactNativeApp
npx expo run:ios
```

This builds the `ReactNativePoC` app target (not the framework) and opens it directly on the simulator. Use this for JS development and native module debugging before publishing the XCFramework.

### Running the native consumer app on the simulator

Open `TescoUIKitApp/TescoUIKitApp.xcodeproj` in Xcode and press Run (⌘R). The app uses the pre-built XCFrameworks in `ReactNativeApp/artifacts/`. The JS bundle must be reachable — either:

- **Development:** Metro is running (`npx react-native start`) and the device/simulator resolves `localhost:8081`
- **Production / standalone test:** The JS bundle is embedded (see [Section 9](#9-building-the-xcframework))

## 5. Adding a New RN Screen

### JS side

Create a new component in `ReactNativeApp/src/`:

```
src/
  App.tsx           ← existing root
  screens/
    ProductScreen.tsx   ← new
    CheckoutScreen.tsx  ← new
```

Register the screen in `index.js` using `AppRegistry`:

```js
// index.js
import { AppRegistry } from 'react-native';
import App from './src/App';
import ProductScreen from './src/screens/ProductScreen';

AppRegistry.registerComponent('TescoRNApp', () => App);
AppRegistry.registerComponent('ProductScreen', () => ProductScreen);
AppRegistry.registerComponent('CheckoutScreen', () => CheckoutScreen);
```

Each registered name becomes a `moduleName` you can pass to the native side.

### Native side (consuming the new screen)

No XCFramework rebuild is needed if no new native modules are introduced. The native team instantiates the new screen by name:

```swift
// SwiftUI
ReactNativeView(moduleName: "ProductScreen", initialProps: ["sku": "abc123"])

// UIKit
ReactNativeViewController(moduleName: "ProductScreen", initialProps: ["sku": "abc123"])
```

If a new native module is required by the screen, follow Section 6 first, rebuild the XCFramework, and publish the new version.

## 6. Adding a New Native Capability

This is the responsibility of the platform team. A native capability is any feature that requires access to iOS APIs not reachable from JS (Keychain, CoreLocation, push notifications, CoreBluetooth, etc.).

### Step 1 — Create the Expo Module

Add a new Swift file to `ReactNativeApp/ios/TescoNativeBridgePod/`:

```swift
// ReactNativeApp/ios/TescoNativeBridgePod/LocationModule.swift
import ExpoModulesCore
import CoreLocation

public class LocationModule: Module {
    public func definition() -> ModuleDefinition {
        Name("TescoLocation")

        // Synchronous property
        Property("isAvailable") {
            CLLocationManager.locationServicesEnabled()
        }

        // Async function callable from JS
        AsyncFunction("requestPermission") { (promise: Promise) in
            // ... request location permission, resolve/reject promise
        }

        // Event emitter — JS can subscribe with addListener("onLocationUpdate", callback)
        Events("onLocationUpdate")
    }
}
```

That is the complete native implementation. No ObjC, no C++, no header files.

### Step 2 — Register the module

Open `ReactNativeApp/ios/TescoNativeBridgePod/ExpoModulesProvider.swift` and add the new module to `getModuleClasses()`:

```swift
public func getModuleClasses() -> [AnyClass] {
    return [
        TescoNativeBridgeModule.self,
        LocationModule.self,   // add this line
    ]
}
```

### Step 3 — Expose the TypeScript interface (optional but recommended)

```ts
// ReactNativeApp/src/modules/TescoLocation.ts
import { requireNativeModule, EventEmitter } from 'expo-modules-core';

const LocationModule = requireNativeModule<{
    isAvailable: boolean;
    requestPermission(): Promise<void>;
}>('TescoLocation');

export const locationEmitter = new EventEmitter(LocationModule);
export default LocationModule;
```

### Step 4 — Expose a Combine publisher in the XCFramework (if native app needs to subscribe)

If the native app (not JS) needs to react to the event, add a publisher to `BridgeEvents.swift`:

```swift
// ReactNativeApp/ios/tescornappbrownfield/BridgeEvents.swift
public enum BridgeEvents {
    public static let buttonTapped: AnyPublisher<Void, Never> = …   // existing

    public static let locationUpdated: AnyPublisher<CLLocation, Never> =
        NotificationCenter.default
            .publisher(for: NSNotification.Name("TescoLocationUpdate"))
            .compactMap { $0.object as? CLLocation }
            .eraseToAnyPublisher()
}
```

### Step 5 — Rebuild and publish the XCFramework

```bash
cd ReactNativeApp
npx expo-brownfield build:ios --release --scheme tescornappbrownfield
```

See [Section 9](#9-building-the-xcframework) for full build instructions and [Section 10](#10-distributing-the-xcframework) for publishing.

## 7. Consuming the Framework in the Native App

### Adding a new RN surface (SwiftUI)

```swift
import SwiftUI
import tescornappbrownfield

struct CheckoutView: View {
    var body: some View {
        ReactNativeView(
            moduleName: "CheckoutScreen",
            initialProps: ["basketId": basketId, "locale": "en-GB"]
        )
    }
}
```

### Adding a new RN surface (UIKit)

```swift
import UIKit
import tescornappbrownfield

let vc = ReactNativeViewController(
    moduleName: "CheckoutScreen",
    initialProps: ["basketId": basketId]
)
navigationController?.pushViewController(vc, animated: true)
```

### Passing state from native to RN

Use `BrownfieldState` for shared key-value state. Set values from native before presenting the RN surface:

```swift
BrownfieldState.set("authToken", token)
BrownfieldState.set("userId", user.id)
```

Read from JS:

```ts
import { BrownfieldState } from 'expo-brownfield';
const token = BrownfieldState.get('authToken');
```

### Subscribing to RN events from native

```swift
import Combine
import tescornappbrownfield

class CheckoutCoordinator {
    private var cancellables = Set<AnyCancellable>()

    func start() {
        BrownfieldMessaging.addListener { message in
            guard let event = message["event"] as? String else { return }
            if event == "checkout_complete" {
                // handle checkout completion
            }
        }
    }
}
```

### Initialising the RN runtime

Call this once, as early as possible in the app lifecycle (typically `AppDelegate`):

```swift
func application(_ application: UIApplication,
                 didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
    ReactNativeHostManager.shared.initialize()
    // …
    return true
}
```

`initialize()` is idempotent. Calling it at launch pre-warms Hermes, reducing the latency of the first RN screen.

## 8. Testing

### JS unit tests

Add Jest to the project:

```bash
cd ReactNativeApp
npm install --save-dev jest @testing-library/react-native @types/jest babel-jest
```

Add to `package.json`:

```json
{
  "scripts": {
    "test": "jest"
  },
  "jest": {
    "preset": "react-native",
    "setupFilesAfterFramework": ["@testing-library/react-native/extend-expect"]
  }
}
```

Write tests alongside components:

```ts
// src/screens/__tests__/ProductScreen.test.tsx
import React from 'react';
import { render, screen } from '@testing-library/react-native';
import ProductScreen from '../ProductScreen';

it('renders the product name', () => {
    render(<ProductScreen sku="abc123" />);
    expect(screen.getByText('Product abc123')).toBeTruthy();
});
```

Run:

```bash
cd ReactNativeApp
npm test
```

### TypeScript type-checking

```bash
cd ReactNativeApp
npx tsc --noEmit
```

### Native unit tests (XCTest)

The `ReactNativePoC.xcworkspace` (RN side) has a test target. Write tests for native modules using XCTest:

```swift
// ReactNativeAppTests/TescoNativeBridgeTests.swift
import XCTest
@testable import TescoNativeBridge

class TescoNativeBridgeTests: XCTestCase {
    func testButtonTappedPostsNotification() throws {
        let exp = expectation(forNotification: .init("TescoNativeBridgeButtonTapped"),
                              object: nil)
        // Call the module function directly (without going through JS)
        NotificationCenter.default.post(name: .init("TescoNativeBridgeButtonTapped"),
                                        object: "test")
        wait(for: [exp], timeout: 1)
    }
}
```

For the consumer app (`TescoUIKitApp`), test the Combine layer and SwiftUI view models in isolation. `CartState` can be tested without RN:

```swift
// TescoUIKitAppTests/CartStateTests.swift
import XCTest
import Combine
@testable import TescoUIKitApp

class CartStateTests: XCTestCase {
    var cancellables = Set<AnyCancellable>()

    func testCountIncrementsOnNotification() {
        let cart = CartState.shared
        let initial = cart.count

        NotificationCenter.default.post(
            name: .init("TescoNativeBridgeButtonTapped"),
            object: nil
        )

        let exp = XCTestExpectation(description: "count increments")
        cart.$count
            .dropFirst()
            .sink { count in
                XCTAssertEqual(count, initial + 1)
                exp.fulfill()
            }
            .store(in: &cancellables)

        wait(for: [exp], timeout: 1)
    }
}
```

### End-to-end tests

[Maestro](https://maestro.mobile.dev) is recommended for E2E — it drives the simulator via accessibility IDs with no instrumentation changes required in production code.

Install:

curl -Ls "https://get.maestro.mobile.dev" | bash

## 9. Building the XCFramework

Run from `ReactNativeApp/`:

```bash
cd ReactNativeApp
npx expo-brownfield build:ios --release --scheme tescornappbrownfield
```

Output:

```
artifacts/
  tescornappbrownfield.xcframework   (~16 MB device slice, ~30 MB simulator slice)
  hermesvm.xcframework               (~4.6 MB device slice, ~9.2 MB simulator slice)
```

### When to rebuild

| Change | Requires XCFramework rebuild? |
|---|---|
| JS / TypeScript change | No — only JS bundle changes |
| New or modified Expo Module (Swift) | Yes |
| New npm package with native code | Yes |
| Expo SDK upgrade | Yes |
| BridgeEvents / BrownfieldState API change | Yes |
| RN version upgrade | Yes |

### Building for a specific configuration

```bash
# Debug build (includes dev menu, source maps — not for production)
npx expo-brownfield build:ios --scheme tescornappbrownfield

# Release build (dead code stripped, no dev menu)
npx expo-brownfield build:ios --release --scheme tescornappbrownfield
```

Always use `--release` for XCFrameworks distributed to the native app team or published to CI.

### Bundling the JS with the XCFramework build

By default, the XCFramework loads JS from Metro in development and from an embedded bundle in release. To embed the JS bundle inside the native app's main bundle, build the bundle separately:

```bash
cd ReactNativeApp
npx react-native bundle \
  --platform ios \
  --dev false \
  --entry-file index.js \
  --bundle-output ../TescoUIKitApp/TescoUIKitApp/main.jsbundle \
  --assets-dest ../TescoUIKitApp/TescoUIKitApp
```

Then add `main.jsbundle` to the `TescoUIKitApp` Xcode target. In production this is handled by CI (see [Section 12](#12-ci-pipeline)).

## 10. Distributing the XCFramework

### Current (PoC): committed to the repository

The `artifacts/` directory is committed. This works at small scale but does not version the framework independently of the JS codebase.

### Recommended: binary store with versioned releases

Publish the XCFrameworks to an internal binary store (S3, Artifactory, GitHub Releases) and reference them by version.

**Step 1 — Tag a release and upload**

```bash
VERSION="1.2.0"
tar -czf tescornappbrownfield-${VERSION}.tar.gz -C ReactNativeApp/artifacts .
# Upload to your binary store, e.g.:
aws s3 cp tescornappbrownfield-${VERSION}.tar.gz s3://tesco-ios-frameworks/rn/${VERSION}/
```

**Step 2 — Reference in the native app (Tuist)**

```swift
// Project.swift (Tuist)
let rnFramework = TargetDependency.xcframework(
    path: .relativeToRoot("vendor/rn/tescornappbrownfield.xcframework")
)
```

Or via SPM binary target once the team moves to fully SPM-based distribution:

```swift
// Package.swift
.binaryTarget(
    name: "tescornappbrownfield",
    url: "https://artifacts.tesco.internal/rn/1.2.0/tescornappbrownfield.xcframework.zip",
    checksum: "abc123…"
)
```

The consumer API (`ReactNativeView`, `BridgeEvents`, `BrownfieldState`) is stable across framework versions as long as no breaking API changes are made to the Swift public interface. Version according to semantic versioning: breaking public API changes increment the major version.

## 11. OTA Updates with EAS

OTA (over-the-air) updates allow JS bundle changes to be deployed without an App Store release. This is the primary value of using Expo in a brownfield context.

### Setup

```bash
cd ReactNativeApp
npm install --save-dev eas-cli
npx eas login
npx eas update:configure
```

This creates `eas.json`:

```json
{
  "cli": { "version": ">= 12.0.0" },
  "build": {
    "development": { "distribution": "internal" },
    "production": { "distribution": "store" }
  },
  "update": {
    "channel": "production"
  }
}
```

### Publishing an update

```bash
# Publish a JS-only change to all users on the production channel
npx eas update --channel production --message "Fix checkout price display"
```

The native app will fetch and apply the new bundle on the next foreground transition (configurable).

### What can and cannot be OTA updated

| Change type | OTA deployable | Notes |
|---|---|---|
| UI changes (components, styles) | Yes | |
| Business logic | Yes | |
| New JS screens registered in AppRegistry | Yes | |
| New npm packages (JS only) | Yes | |
| New native module (Expo Module) | No | Requires XCFramework rebuild + app update |
| RN / Expo SDK upgrade | No | Requires XCFramework rebuild + app update |
| Changes to `BridgeEvents` / `BrownfieldState` Swift API | No | Requires XCFramework rebuild + app update |

### Channels strategy for Tesco

```
main  →  development channel  (internal team testing)
staging → staging channel     (QA / release candidate)
main   → production channel   (live users, after App Store approval)
```

OTA updates on the `production` channel reach all users who have the current app version installed. A rollback is as simple as:

```bash
npx eas update --channel production --rollback-to-commit <sha>
```

## 12. CI Pipeline

Two independent pipelines run in parallel.

### Pipeline A — XCFramework build (platform team, triggers on native module changes)

```yaml
# .github/workflows/xcframework.yml
name: Build XCFramework

on:
  push:
    paths:
      - 'ReactNativeApp/ios/TescoNativeBridgePod/**'
      - 'ReactNativeApp/ios/tescornappbrownfield/**'
      - 'ReactNativeApp/package.json'

jobs:
  build:
    runs-on: macos-15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install JS deps
        run: cd ReactNativeApp && npm ci
      - name: Install pods
        run: cd ReactNativeApp/ios && pod install
      - name: Build XCFramework
        run: cd ReactNativeApp && npx expo-brownfield build:ios --release --scheme tescornappbrownfield
      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: xcframeworks
          path: ReactNativeApp/artifacts/
      - name: Publish to binary store
        run: |
          VERSION=$(node -p "require('./ReactNativeApp/package.json').version")
          # upload artifacts to internal store at $VERSION
```

### Pipeline B — JS bundle tests and OTA publish (RN team, triggers on src/ changes)

```yaml
# .github/workflows/rn-js.yml
name: RN JS — Test & Deploy

on:
  push:
    paths:
      - 'ReactNativeApp/src/**'
      - 'ReactNativeApp/index.js'

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: cd ReactNativeApp && npm ci
      - run: cd ReactNativeApp && npm test
      - run: cd ReactNativeApp && npx tsc --noEmit

  deploy-ota:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: cd ReactNativeApp && npm ci
      - run: cd ReactNativeApp && npx eas update --channel production --non-interactive
        env:
          EXPO_TOKEN: ${{ secrets.EXPO_TOKEN }}
```

### Pipeline C — Native consumer app build (native iOS team, triggers on XCFramework update or TescoUIKitApp changes)

```yaml
# .github/workflows/native-app.yml
name: Native App Build

on:
  push:
    paths:
      - 'TescoUIKitApp/**'
      - 'ReactNativeApp/artifacts/**'   # or triggered by XCFramework publish event

jobs:
  build:
    runs-on: macos-15
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: |
          xcodebuild \
            -project TescoUIKitApp/TescoUIKitApp.xcodeproj \
            -scheme TescoUIKitApp \
            -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
            build
```

## 13. Scaling to Multiple Features and Teams

### JS codebase structure

As feature count grows, organise the JS source by product domain rather than by technical type:

```
src/
  features/
    checkout/
      CheckoutScreen.tsx
      CheckoutScreen.test.tsx
      api.ts
      types.ts
    loyalty/
      LoyaltyScreen.tsx
      …
    recipes/
      RecipesScreen.tsx
      …
  shared/
    components/
    hooks/
    bridge/           ← typed wrappers around native modules
      TescoNativeBridge.ts
      TescoLocation.ts
  App.tsx
index.js
```

Keep `index.js` as the only file that calls `AppRegistry.registerComponent`. Each new screen is one `registerComponent` call.

### Single RN instance — coordination model

All RN screens share one JS runtime. This means:

- One `package.json` — all native module dependencies are declared together
- One Metro bundle — all screens are in the same bundle
- One Expo SDK version — all features upgrade together

**For one or two teams:** a single repo and a single bundle is the right approach. The coordination overhead is low.

**For three or more teams with independent release cadences:** investigate [Re.Pack](https://re-pack.dev) (Webpack-based bundler that supports Module Federation) or [Metro bundle splitting](https://metrobundler.dev/docs/configuration/#serializer-options). This allows each team to own a separate chunk loaded lazily, while still sharing one RN runtime and one XCFramework.

### Native module governance

Establish a clear ownership model early:

- `TescoNativeBridgePod/` is owned by the **platform team**
- Any new module request from a feature team goes through a lightweight RFC: module name, JS interface, iOS API used
- The platform team reviews, implements, and cuts a new XCFramework version
- Feature teams never add to `TescoNativeBridgePod/` directly

This keeps the XCFramework's public API coherent and prevents ad-hoc proliferation of modules.

### BrownfieldState and BrownfieldMessaging contracts

Document the keys used in `BrownfieldState` and the event shapes in `BrownfieldMessaging` in a shared contract file. Treat them as an API — breaking changes require a major version bump:

```ts
// src/bridge/contracts.ts
export const StateKeys = {
    AUTH_TOKEN: 'authToken',
    USER_ID: 'userId',
    BASKET_ID: 'basketId',
} as const;

export type BridgeEvent =
    | { event: 'checkout_complete'; orderId: string }
    | { event: 'loyalty_card_scanned'; cardNumber: string };
```

## 14. Troubleshooting

### XCFramework build fails with hermesc path error

**Symptom:** `hermesc: no such file or directory at path ReactNativePoC/node_modules/...`

The CocoaPods-generated xcconfig has a stale absolute path from before a directory rename.

```bash
cd ReactNativeApp/ios
rm -rf Pods Podfile.lock build
pod install
```

### PCH (precompiled header) module cache mismatch

**Symptom:** `error: PCH file ... was compiled with module cache path ... which is not the module cache path`

```bash
rm -rf ReactNativeApp/ios/build
```

Then rebuild.

### `Cannot find 'BridgeEvents' in scope` in consumer app

**Symptom:** After adding a new file to the framework sources, it is not visible to the consumer.

The file must be registered in `ReactNativePoC.xcodeproj/project.pbxproj` for the `tescornappbrownfield` target. Open the Xcode project, drag the file into the `tescornappbrownfield` group (not just the folder), make sure `tescornappbrownfield` is checked in the Target Membership panel. Then rebuild the XCFramework.

### `Unable to find module dependency: 'TescoNativeBridge'` in swiftinterface

**Symptom:** Consumer app fails with this linker/module error after XCFramework build.

A `public import TescoNativeBridge` in a framework source file leaked into the public `.swiftinterface`. Change it to `internal import TescoNativeBridge`. Rebuild the XCFramework.

### Metro cannot connect in the simulator

Ensure Metro is running (`npx react-native start`) and that the simulator can reach `localhost:8081`. In Xcode 15+, the simulator may require explicit network access. In `Info.plist` of the RN app target, ensure:

```xml
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSAllowsLocalNetworking</key>
    <true/>
</dict>
```

### RN screen is blank on first launch

`ReactNativeHostManager.shared.initialize()` was not called early enough. Call it in `AppDelegate.application(_:didFinishLaunchingWithOptions:)` before creating any view.

### Pod install fails with Expo module headers not found

```bash
cd ReactNativeApp/ios
pod deintegrate
pod install
```

If it persists, delete the Xcode DerivedData: `rm -rf ~/Library/Developer/Xcode/DerivedData`.

## 15. Dependency Version Policy

The three layers must be kept in sync. Mismatched versions between these are the primary source of build failures.

| Dependency | Version | Upgrade policy |
|---|---|---|
| `react-native` | 0.83.2 | Upgrade with Expo SDK — do not upgrade independently |
| `expo` | ~55.0.0 | Upgrade on each Expo SDK release (~3 months) |
| `expo-brownfield` | ~55.0.0 | Must match `expo` major version exactly |
| `expo-modules-core` | ^55.0.12 | Follows `expo` major version |
| Hermes | bundled with RN | Never configure separately |
| Node.js | ≥ 20.19.4 | LTS only |
| CocoaPods | ≥ 1.15 | Upgrade when pod install reports incompatibility |

**Expo SDK upgrade checklist (every ~3 months):**

1. `npm install expo@<new> expo-brownfield@<new> expo-modules-core@<new>`
2. Check the Expo SDK changelog for breaking changes in `expo-brownfield`
3. Update `react-native` to the version targeted by the new SDK
4. `cd ios && pod install`
5. Run `npx expo-brownfield build:ios --release --scheme tescornappbrownfield`
6. Smoke test on simulator
7. Publish new XCFramework version
8. Update native app's XCFramework reference

Do not fall behind by more than one Expo SDK major. Expo drops support for SDK N-1 approximately 12 months after SDK N ships. Upgrades that span two major versions are significantly more complex.
