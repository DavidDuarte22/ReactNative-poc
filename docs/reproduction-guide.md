# Reproduction Guide — React Native Brownfield XCFramework PoC

This guide documents exactly what was done to reach the current working state, including every
workaround, why it was needed, and what to validate when reproducing from scratch. Items marked
**[IMPROVE?]** are candidates for simplification in a future iteration.

---

## What you will end up with

```
ReactNativePoC/
├── ReactNativeApp/                     ← Expo project, builds XCFrameworks
│   ├── ios/
│   │   ├── TescoNativeBridgePod/       ← one Swift file per native capability
│   │   ├── tescornappbrownfield/       ← framework shell (generated by expo prebuild)
│   │   ├── Podfile                     ← manually maintained (no use_expo_modules!)
│   │   └── ReactNativePoC.xcworkspace  ← used by expo-brownfield build only
│   ├── src/App.tsx
│   ├── artifacts/
│   │   ├── tescornappbrownfield.xcframework   (~21 MB device slice)
│   │   └── hermesvm.xcframework               (~4.6 MB device slice)
│   └── package.json
└── TescoUIKitApp/
    ├── TescoUIKitApp.xcodeproj          ← no CocoaPods, no Node required
    └── TescoUIKitApp/
        ├── AppDelegate.swift
        ├── HomeView.swift               ← SwiftUI, embeds ReactNativeView
        └── CartState.swift              ← Combine, subscribes to BridgeEvents
```

## Phase 1 — Create the React Native project

> **Version constraint (critical):** Expo SDK 55 targets React Native 0.83.2 exactly. Do NOT use
> RN 0.84+ — it added a `bundleConfiguration` parameter to `view()` that breaks the Expo SDK 55
> Swift code inside the framework template.

```bash
mkdir ReactNativePoC && cd ReactNativePoC
mkdir ReactNativeApp && cd ReactNativeApp

# Initialise with RN 0.83.2 specifically
npx @react-native-community/cli@19.1.2 init ReactNativePoC \
  --version 0.83.2 \
  --directory . \
  --skip-install
```

> `--directory .` puts the project files directly into `ReactNativeApp/` rather than a
> subdirectory. `--skip-install` lets you control `npm install` manually.

Verify:

```bash
cat package.json | grep '"react-native"'
# should output: "react-native": "0.83.2"
```

## Phase 2 — Install Expo and expo-brownfield

```bash
# Still in ReactNativeApp/
npm install

npm install expo@~55.0.0 expo-brownfield@~55.0.0 expo-modules-core@^55.0.12
```

Configure `app.json` to declare the `expo-brownfield` plugin and a valid Expo slug:

```json
{
  "expo": {
    "name": "TescoRNApp",
    "slug": "tesco-rn-app",
    "version": "0.1.0",
    "platforms": ["ios"],
    "ios": {
      "bundleIdentifier": "com.yourorg.rnpoc.brownfield"
    },
    "plugins": [
      "expo-brownfield"
    ]
  }
}
```

Configure `babel.config.js` to inline `process.env.EXPO_OS` (required by expo-modules-core JS
code, which expects `babel-preset-expo` to replace this — we are not using that preset):

```js
// babel.config.js
function inlineExpoOS() {
  return {
    visitor: {
      MemberExpression(path) {
        if (
          path.get('object').matchesPattern('process.env') &&
          path.node.property.name === 'EXPO_OS'
        ) {
          path.replaceWith({ type: 'StringLiteral', value: 'ios' });
        }
      },
    },
  };
}

module.exports = {
  presets: ['module:@react-native/babel-preset'],
  plugins: [inlineExpoOS],
  parserOpts: {
    enableExperimentalFlowMatchSyntax: true,
  },
};
```

> `enableExperimentalFlowMatchSyntax: true` is required because RN 0.83.x uses Flow `match`
> expressions in some internal files that Hermes parser needs a flag to handle.

`expo-brownfield build:ios` will compile into the XCFramework. It also triggers the

## Phase 4 — Apply workarounds

These are the non-obvious steps that were required to make the build work. Each is documented
with the root cause so you can validate whether it is still needed when reproducing.

---

### Workaround 1 — Inject "Bundle React Native code and images" build phase

**Root cause:** `expo-brownfield`'s config plugin checks for a `PBXShellScriptBuildPhase` named
`"Bundle React Native code and images"` in the main app target before creating the framework
target. This phase is created automatically by `react-native init` in a standard setup, but was
absent from the hand-crafted project (or not created by `expo prebuild`).

**Symptom:** `expo-brownfield build:ios` exits with an error about a missing build phase, or the
framework target is created without the JS bundling step.

**Fix:** Open `ios/ReactNativePoC.xcodeproj/project.pbxproj` and:

1. Add this entry to the `PBXShellScriptBuildPhase` section:

```
C3D4E5F6A7B8C90AB12C3D4E /* Bundle React Native code and images */ = {
    isa = PBXShellScriptBuildPhase;
    buildActionMask = 2147483647;
    files = ();
    inputFileListPaths = ();
    inputPaths = ();
    name = "Bundle React Native code and images";
    outputFileListPaths = ();
    outputPaths = ();
    runOnlyForDeploymentPostprocessing = 0;
    shellPath = /bin/sh;
    shellScript = "set -e\n\nWITH_ENVIRONMENT=\"${REACT_NATIVE_PATH}/scripts/xcode/with-environment.sh\"\nREACT_NATIVE_XCODE=\"${REACT_NATIVE_PATH}/scripts/react-native-xcode.sh\"\n\n/bin/sh -c \"$WITH_ENVIRONMENT \\\"$REACT_NATIVE_XCODE\\\"\"\n";
};
```

2. Add `C3D4E5F6A7B8C90AB12C3D4E /* Bundle React Native code and images */,` to the
   `buildPhases` array of the `ReactNativePoC` app target.

**[IMPROVE?]** Starting from a fresh `react-native init` (without a hand-crafted project) this
build phase will already exist. If CNG (`expo prebuild`) is used from the start, verify it is
created automatically. If so, this workaround is unnecessary for a greenfield setup.

---

### Workaround 2 — Do NOT call `use_expo_modules!` in the Podfile

**Root cause:** `use_expo_modules!` auto-generates an `ExpoModulesProvider.swift` file in the
project. We maintain our own `ExpoModulesProvider.swift` inside `TescoNativeBridgePod/` (required
to register custom modules). Having two files with the same `@objc(ExpoModulesProvider)` class
causes a duplicate symbol linker error.

**Fix:** Never call `use_expo_modules!`. Install ExpoModulesCore and ExpoModulesJSI as explicit
pod declarations:

```ruby
pod 'ExpoModulesJSI', :path => '../node_modules/expo-modules-core'
pod 'ExpoModulesCore', :path => '../node_modules/expo-modules-core', :modular_headers => true
pod 'ExpoBrownfield', :path => '../node_modules/expo-brownfield/ios'
pod 'EXManifests', :path => '../node_modules/expo-manifests/ios'
```

The `:modular_headers => true` on `ExpoModulesCore` is required so Swift can import it as a
module (Swift `import ExpoModulesCore` depends on the module map that modular_headers generates).

**[IMPROVE?]** Investigate whether `use_expo_modules!` can be called with an option to skip
generating `ExpoModulesProvider.swift`. If that option exists in a future Expo SDK version, the
manual pod declarations could be replaced.

---

### Workaround 3 — Add ExpoBrownfield and EXManifests pods explicitly

**Root cause:** The `tescornappbrownfield/` framework template imports `ExpoBrownfield` and
`EXManifests` in its Swift files. These are not auto-installed by `expo-brownfield build:ios`
when `use_expo_modules!` is absent.

**Fix:** Already included in workaround 2 above. Both pods must be listed explicitly:

```ruby
pod 'ExpoBrownfield', :path => '../node_modules/expo-brownfield/ios'
pod 'EXManifests', :path => '../node_modules/expo-manifests/ios'
```

---

### Workaround 4 — post_install: inject ExpoModulesCore headers into the Expo pod

**Root cause:** Without `use_expo_modules!`, CocoaPods does not add ExpoModulesCore or
ExpoModulesJSI to the `Expo` pod's xcconfig. The `Expo` pod's `.mm` files include
`<ExpoModulesCore/ExpoModulesCore.h>` and need the module maps to resolve. It also needs
explicit build order: `ExpoModulesCore` must compile before `Expo` so that
`ExpoModulesCore-Swift.h` (the ObjC bridge header, generated at compile time) exists.

**Fix:** Add to the `post_install` block in the Podfile:

```ruby
post_install do |installer|
  react_native_post_install(installer, config[:reactNativePath], :mac_catalyst_enabled => false)

  expo_target      = installer.pods_project.targets.find { |t| t.name == 'Expo' }
  expo_core_target = installer.pods_project.targets.find { |t| t.name == 'ExpoModulesCore' }
  expo_jsi_target  = installer.pods_project.targets.find { |t| t.name == 'ExpoModulesJSI' }

  # Enforce build order — ExpoModulesCore must compile before Expo
  expo_target.add_dependency(expo_core_target) if expo_target && expo_core_target
  expo_target.add_dependency(expo_jsi_target)  if expo_target && expo_jsi_target

  expo_core_headers    = '"${PODS_ROOT}/Headers/Public/ExpoModulesCore"'
  expo_core_modulemap  = '"${PODS_ROOT}/Headers/Public/ExpoModulesCore/ExpoModulesCore.modulemap"'
  expo_jsi_headers     = '"${PODS_ROOT}/Headers/Public/ExpoModulesJSI"'
  expo_jsi_modulemap   = '"${PODS_ROOT}/Headers/Public/ExpoModulesJSI/ExpoModulesJSI.modulemap"'
  expo_core_swift_mod  = '"${PODS_CONFIGURATION_BUILD_DIR}/ExpoModulesCore"'

  installer.pods_project.targets.each do |target|
    next unless target.name == 'Expo'
    target.build_configurations.each do |bc|
      bc.build_settings['HEADER_SEARCH_PATHS'] =
        "$(inherited) #{expo_core_headers} #{expo_jsi_headers}"
      bc.build_settings['OTHER_CFLAGS'] =
        "$(inherited) -fmodule-map-file=#{expo_core_modulemap} -fmodule-map-file=#{expo_jsi_modulemap}"
      bc.build_settings['OTHER_SWIFT_FLAGS'] =
        "$(inherited) -Xcc -fmodule-map-file=#{expo_core_modulemap} -Xcc -fmodule-map-file=#{expo_jsi_modulemap}"
      bc.build_settings['SWIFT_INCLUDE_PATHS'] =
        "$(inherited) #{expo_core_swift_mod}"
    end
  end
end
```

**[IMPROVE?]** This is the most fragile part of the setup. Each Expo SDK upgrade risks breaking
these path assumptions. A cleaner approach would be to patch the `Expo.podspec` upstream, or to
use `use_expo_modules!` with a suppressed provider (if that becomes available). Track this on
each Expo SDK upgrade.

require "json"
package = JSON.parse(File.read(File.join(__dir__, "package.json")))

Pod::Spec.new do |s|
  s.name         = "TescoNativeBridge"
  s.version      = package["version"]
  s.summary      = package["description"]
  s.homepage     = "https://github.com/tesco/rn-poc"
  s.license      = "MIT"

**`TescoNativeBridgePod/ExpoModulesProvider.swift`**

This file is required by the Expo runtime to discover registered modules at startup. The class
name `ExpoModulesProvider` must match exactly (including the `@objc` annotation) — it is found
at runtime via `NSClassFromString("ExpoModulesProvider")`.

```swift
import ExpoModulesCore

@objc(ExpoModulesProvider)
public class ExpoModulesProvider: ModulesProvider {
  public override func getModuleClasses() -> [ExpoModuleTupleType] {
    return [
      (TescoNativeBridgeModule.self, nil),
    ]
  }
}
```

**`TescoNativeBridgePod/TescoNativeBridgeModule.swift`**

```swift
import ExpoModulesCore

public class TescoNativeBridgeModule: Module {
  public func definition() -> ModuleDefinition {
    Name("TescoNativeBridge")

    AsyncFunction("onButtonTapped") { (message: String) in
      NotificationCenter.default.post(
        name: .init("TescoNativeBridgeButtonTapped"),
        object: message
      )
    }
  }
}
```

## Phase 7 — Add BridgeEvents to the framework sources

`BridgeEvents.swift` is the public Combine interface of the XCFramework. It lives in
`ios/tescornappbrownfield/` (alongside the generated framework sources) and must be registered
in the Xcode project to be compiled into the framework.

**Create the file:**

```swift
// ReactNativeApp/ios/tescornappbrownfield/BridgeEvents.swift
import Combine
import Foundation

public enum BridgeEvents {
    public static let buttonTapped: AnyPublisher<Void, Never> =
        NotificationCenter.default
            .publisher(for: NSNotification.Name("TescoNativeBridgeButtonTapped"))
            .map { _ in () }
            .eraseToAnyPublisher()
}
```

**Register it in the Xcode project (required — the "Recovered References" group does not
auto-discover files):**

Open `ReactNativeApp/ios/ReactNativePoC.xcodeproj` in Xcode, navigate to the
`tescornappbrownfield` group in the Project Navigator, and drag `BridgeEvents.swift` into it.
In the Target Membership inspector on the right, ensure only `tescornappbrownfield` is checked
(not `ReactNativePoC`).

Alternatively, edit `project.pbxproj` directly following the pattern of `Messaging.swift`:

1. Add a `PBXFileReference` entry with `name = BridgeEvents.swift; path = tescornappbrownfield/BridgeEvents.swift`
2. Add the UUID to the `"Recovered References"` group's `children` array
3. Add a `PBXBuildFile` entry referencing the file ref
4. Add the `PBXBuildFile` UUID to the `57A4D3A14BA140508845E523` (`tescornappbrownfield`) Sources build phase

**[IMPROVE?]** This manual registration step exists because the framework template's source
directory is not set up as a `PBXFileSystemSynchronizedRootGroup` (which would auto-discover
files). Setting the `tescornappbrownfield/` directory as a synchronized group in Xcode would
eliminate the need to manually edit `project.pbxproj` every time a new Swift file is added to
the framework.

**Workaround 5 — use `internal import` for any pod imports in framework sources:**

In `ReactNativeHostManager.swift` (generated by expo prebuild), change:

```swift
// Before — leaks TescoNativeBridge into the public .swiftinterface
import TescoNativeBridge

// After — internal only, not visible to consumers
internal import TescoNativeBridge
```

**Root cause:** With `-enable-library-evolution` (required for XCFramework binary stability),
any bare `import` in a framework source file appears in the public `.swiftinterface`. Consumers
of the XCFramework then need `TescoNativeBridge` as a resolvable module, which it is not — it
is a pod internal to the build, not shipped to consumers.

**Symptom:** `Unable to find module dependency: 'TescoNativeBridge'` when building the consumer
app.

**Rule:** Any `import` of a pod-only module inside `ios/tescornappbrownfield/` must be declared
`internal import`. Only framework types visible to consumers may be `public import`.

## Phase 9 — Create the native consumer app (TescoUIKitApp)

Create a new Xcode project: **File → New → Project → App**, UIKit, Swift, no StoryBoard (or use
SwiftUI lifecycle). Name it `TescoUIKitApp`.

### Add the XCFrameworks

In Xcode: select the project → target `TescoUIKitApp` → **General → Frameworks, Libraries, and
Embedded Content** → click `+` → **Add Other → Add Files** → navigate to
`ReactNativeApp/artifacts/` and add both:

- `tescornappbrownfield.xcframework` — set to **Embed & Sign**
- `hermesvm.xcframework` — set to **Embed & Sign**

### Disable sandboxing for script phases

In **Build Settings**, set `ENABLE_USER_SCRIPT_SANDBOXING = NO`. Required for CocoaPods-style
rsync script phases inside the XCFrameworks.

### AppDelegate.swift

```swift
import UIKit
import SwiftUI
import tescornappbrownfield

@main
class AppDelegate: UIResponder, UIApplicationDelegate {
    var window: UIWindow?

    func application(_ application: UIApplication,
                     didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        // Boot the Combine observer before any RN surface can fire events
        _ = CartState.shared
        // Pre-warm the Hermes runtime to reduce first-screen latency
        ReactNativeHostManager.shared.initialize()

        window = UIWindow(frame: UIScreen.main.bounds)
        window?.rootViewController = UIHostingController(rootView: HomeView())
        window?.makeKeyAndVisible()
        return true
    }
}
```

### CartState.swift

```swift
import Combine
import Foundation
import tescornappbrownfield

final class CartState: ObservableObject {
    static let shared = CartState()

    @Published private(set) var count = 0
    private var cancellable: AnyCancellable?

    private init() {
        cancellable = BridgeEvents.buttonTapped
            .receive(on: DispatchQueue.main)
            .sink { [weak self] _ in self?.count += 1 }
    }
}
```

> `import Foundation` is required explicitly in Swift 6 — `DispatchQueue` is not transitively
> imported via `Combine` alone.

### HomeView.swift

## What to investigate for improvement

These are the areas with the highest potential for simplification, ranked by impact:

### 1. Can `expo prebuild` replace the manual pbxproj bundle phase injection?

In a greenfield setup that uses CNG (Continuous Native Generation) — where the `ios/` directory
is fully managed by `expo prebuild` — the bundle script build phase should be created
automatically. The manual injection was only needed because we started from a hand-crafted
project. **Test:** run `npx expo prebuild --platform ios` on a fresh directory and check whether
the build phase is present before running `expo-brownfield build:ios`.

### 2. Can the `post_install` Expo header injection be simplified?

The post_install hook is fragile — it depends on exact header path conventions that may shift
between Expo SDK versions. **Test:** after each Expo SDK upgrade, verify `pod install` still
succeeds without modifying the hook. Investigate whether the Expo team plans to add
`use_expo_modules!`-independent support to the `Expo` podspec.

### 3. Can the `tescornappbrownfield/` group be a synchronized root group?

If the `ios/tescornappbrownfield/` directory were configured as a
`PBXFileSystemSynchronizedRootGroup` in Xcode, new Swift files added to the directory would be
auto-discovered without editing `project.pbxproj`. **Test:** convert the group in Xcode
(right-click → Convert to Synchronized Group) and verify the build still works.

### 4. Is RN 0.83.2 still the right version with Expo SDK 55?

Expo SDK 55 targets RN 0.83.2. Expo SDK 56 (when released) will target a newer RN version and
may resolve the `bundleConfiguration` breakage that forced the 0.83.2 pin. **Watch:** Expo SDK
56 release notes and update the RN version constraint accordingly.

### 5. Can the XCFramework size be reduced?

The current device slice is ~21 MB (tescornappbrownfield ~16 MB + hermesvm ~4.6 MB). The
hermesvm framework is the JS engine and cannot be reduced. For the framework binary, measure
which pods contribute most:

```bash
# Check which symbols are largest in the framework binary
nm -size-sort ReactNativeApp/artifacts/tescornappbrownfield.xcframework/ios-arm64/\
tescornappbrownfield.framework/tescornappbrownfield | tail -20
```

Candidates for size reduction: unused `React-Core` subspec headers, Expo modules included by
default that the product does not use.
